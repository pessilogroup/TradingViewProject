# WEEX Integration Master Plan

## 1. Introduction
The objective of the WEEX exchange integration is to expand the system's algorithmic execution capabilities to support WEEX spot and contract trading. This integration enables automated order routing, smart order placement, and risk management based on webhook alerts received from TradingView. The scope of this integration includes:
- Secure connection management and credential handling.
- Dynamic symbol discovery and routing mapping (e.g., routing standard ticker symbols to WEEX contract codes such as `BTCUSDT_UMCBL`).
- Real-time candle scanning using CCXT and direct REST APIs.
- Comprehensive order lifecycle management, including entry order placement and simulated OCO (One-Cancels-the-Other) exits (Stop-Loss and Take-Profit).
- Operational risk gates, including position sizing, drawdown limits, failover behaviors, and alerting.

## 3. Mainnet Deployment Strategy (Option 2)
To transition the WEEX integration safely to mainnet production, the system must adhere to a strict staging procedure, strict safety boundaries, and clear error recovery rules.

### 3.1 Credential Setup and Configuration
Production credentials must be stored securely. 
- Environment variables must be appended to the local `.env` configuration file.
- **Strict Security Mandate**: Production credentials must never be committed to the Git repository. The `.gitignore` file must explicitly exclude `.env` files.

Add the following keys to your production `.env` configuration:
```bash
WEEX_API_KEY="your_production_api_key_here"
WEEX_API_SECRET="your_production_api_secret_here"
WEEX_PASSPHRASE="your_production_passphrase_here"
WEEX_TESTNET=False
WEEX_DRY_RUN=False
```

### 3.2 Safety Limits and Drawdowns
Before enabling live execution, hardcoded risk checks must be enforced to guard against algorithmic failures or erratic market conditions:
- **Minimal Trade Size**: Order size must be restricted to a micro-volume minimum, e.g., `0.001 BTC` or `0.01 ETH`. The equivalent order value must reside in the range of `$5.00` to `$10.00 USDT`.
- **Daily Loss Cap**: Daily losses are capped at a maximum of `$10.00` to `$20.00 USD`. If total losses within a rolling 24-hour window exceed this threshold, the execution engine must reject all new entry alerts.
- **Max Drawdown**: Trailing drawdown is restricted to `5%` of the total account balance. Once the trailing drawdown exceeds this limit, trading activities will pause until manually re-enabled.

### 3.3 Failover and Rejection Handling
Robust error handling and execution safety are critical during active trading:
- **Connection and Network Issues**: If the WEEX API raises connection timeouts, DNS resolution errors, or HTTP status codes in the `4xx` or `5xx` range:
  - The trade engine must immediately route the trade to alternative exchanges (e.g., Bybit or Binance) using configured fallback rules.
  - The operator must be notified immediately via Telegram alerts and system logs.
- **Rejection Resolution**: If an entry order is rejected by WEEX due to insufficient margin, leverage mismatch, or invalid symbol:
  - The trade engine must instantly cancel any associated pending exit orders or OCO orders.
  - This prevents orphan exit orders from remaining active without an opening position, protecting the account from accidental exposure.
  - The rejection event must be logged with high severity and trigger a Telegram alert.

### 3.4 Verification and Smoke Testing Stages
A phased rollout protocol must be executed to verify live operations without exposing capital to unnecessary risk:

```
┌─────────────────────────────────┐
│     Phase 1: Mainnet Dry-Run    │
│  (WEEX_DRY_RUN=True/Testnet=F)  │
└────────────────┬────────────────┘
                 │ (Verifies Signature & Public Feeds)
                 ▼
┌─────────────────────────────────┐
│   Phase 2: Manual Micro-Order   │
│       (CLI Micro-Vol Exec)      │
└────────────────┬────────────────┘
                 │ (Verifies Real Order Lifecycle)
                 ▼
┌─────────────────────────────────┐
│    Phase 3: Webhook Integration │
│     (End-to-End Smoke Test)     │
└─────────────────────────────────┘
```

- **Phase 1: Mainnet Dry-Run**
  - **Configuration**: Set `WEEX_DRY_RUN=True` and `WEEX_TESTNET=False`.
  - **Goal**: Execute standard trade alert workflows. This verifies that API signatures are correctly generated for private endpoints and that public data feeds parse correctly, without placing actual orders on the exchange.
- **Phase 2: Manual Micro-Volume Order Execution**
  - **Configuration**: Set `WEEX_DRY_RUN=False` and `WEEX_TESTNET=False`.
  - **Goal**: Execute a single manual order of `0.001 BTC` via a command-line script. Verify the filled order details, trade logs, and response times to ensure proper execution on the real order book.
- **Phase 3: Webhook Integration Smoke Test**
  - **Configuration**: Standard Webhook routing configuration.
  - **Goal**: Send a simulated micro TradingView alert payload to the system's webhook listener (`/api/webhook`). Verify that the ingress parses the message correctly, routes it to WEEX, and triggers order execution end-to-end with the specified safety controls.
